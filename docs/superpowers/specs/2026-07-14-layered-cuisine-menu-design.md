# 易食卦「分层菜系食单」设计规格

**日期：** 2026-07-14  
**状态：** 已确认方向，待用户审阅规格  
**范围：** 重构“整饬食单”的分类、筛选和默认菜单；首页起卦流程保持极简，不出现菜系偏好或分类引导。

## 1. 产品原则

1. 用户第一次打开网站，只看见餐别、去处、天气、报数和起卦；不需要先选择国家、菜系或食物种类。
2. 默认起卦候选池以丰富的**已启用中餐**为主。
3. 世界菜以可编辑模板的形式预置在菜单库，但默认停用；只有用户在“整饬食单”中主动启用后，才参加后续餐签。
4. 分类用于“找到、整理、启用”餐品，不能把菜单抽屉变成复杂后台。
5. 所有菜单和启用状态只保存到当前浏览器 `localStorage`；模板不代表任何所在地真实供应。

## 2. 数据模型

每个餐品保留现有字段：`id`、`name`、`source`、`venue`、`meals`、`staple`、`protein`、`vegetable`、`flavor`、`enabled`。

新增四个稳定的分类字段：

```js
{
  cuisineZone: '中国菜',
  cuisine: '川菜',
  courseFamily: '火锅麻辣烫',
  dishType: '麻辣烫'
}
```

字段含义：

- `cuisineZone`：饮食版图，如“中国菜”“东亚料理”“东南亚料理”“欧洲料理”；
- `cuisine`：具体菜系，如“川菜”“粤菜”“日料”“泰餐”；
- `courseFamily`：用户选择时最有用的大类，如“粉面”“盖饭”“砂锅煲仔”“寿司饭团”；
- `dishType`：可筛选的小类，如“麻辣烫”“重庆小面”“照烧饭”。

### 2.1 兼容旧本地菜单

分类字段在存储读取时可缺省。旧版或用户自定义餐品缺少分类时，自动归入：

```js
{
  cuisineZone: '中国菜',
  cuisine: '中式日常',
  courseFamily: '其他中式',
  dishType: '自定义'
}
```

不能因更新分类而清空用户已有菜单、食录或启用状态。

## 3. 分类目录

目录使用固定值，但在新增 / 编辑餐品时提供“自定义”选项，避免用户被限制。

### 3.1 中国菜（默认模板启用）

**菜系：**

- 川菜、粤菜、湘菜、鲁菜；
- 苏菜、浙菜、闽菜、徽菜；
- 京津菜、东北菜、西北菜；
- 云南菜、贵州菜、潮汕菜、客家菜；
- 港式、台式、素食、清真风味、地方小吃、中式日常。

**大类：**

- 早餐早点；
- 粥品、粉面、米线；
- 饺子馄饨、包点面点；
- 米饭盖饭、炒饭焖饭；
- 小炒家常菜、卤味熟食；
- 汤羹炖盅、砂锅煲仔；
- 火锅麻辣烫、烧烤煎炸；
- 早茶点心、甜品饮品；
- 轻食素食。

默认中餐模板至少 **140 项**，按早餐、午餐、晚餐和食堂 / 校外地点完整覆盖；它们默认 `enabled: true`。

### 3.2 世界料理（默认模板停用）

**饮食版图与菜系：**

- 东亚料理：日料、韩餐；
- 东南亚料理：泰餐、越南菜、马来 / 新加坡风味、印尼菜；
- 南亚料理：印度菜；
- 欧洲料理：意餐、法餐、西式简餐；
- 美洲料理：墨西哥菜、美式餐；
- 中东 / 地中海：土耳其、中东风味、地中海料理；
- 跨地域：轻食沙拉、三明治汉堡、咖啡烘焙。

世界料理模板至少 **70 项**，覆盖寿司饭团、日式盖饭、拉面、韩式拌饭、泰式饭面、越南粉、咖喱、意面、披萨、汉堡、墨西哥卷、沙拉、三明治等常见类别；它们默认 `enabled: false`。

默认菜单总量至少 **210 项**。模板是可修改的起点，不会宣称本地必然供应。

## 4. 首页与推荐规则

首页 UI 不新增任何菜系、国家或品类选择器。

`chooseMeal()` 的资格规则仍然是：已启用、对应餐别、地点可用、未被本次拒绝。因而：

- 已启用中餐自然进入默认候选池；
- 默认停用的世界料理完全不会干扰第一次起卦；
- 用户在食单中启用某个日料 / 泰餐 / 轻食模板后，该餐品自动和其他已启用餐品一样参加后续起卦；
- 不新增首页偏好，不改变梅花数时合参、去重、均衡或天气逻辑。

## 5. “整饬食单”交互

菜单抽屉顶部在说明下方增加一个紧凑的“食单筛选”区域，不做四层树或侧边后台。

### 5.1 可见控件

1. **范围**：全部 / 中国菜 / 世界料理；
2. **菜系**：随范围变化，仅显示该范围的菜系；
3. **大类**：随范围与菜系变化，仅显示存在餐品的大类；
4. **状态**：全部 / 已启用 / 未启用；
5. **清除筛选**：一键恢复全部。

`dishType` 不再独占一个额外下拉框，以避免复杂度；它显示在每个餐品卡的路径标签中，并在“菜系 + 大类”筛选后通过卡片列表自然呈现。用户要找更细小类时，可用抽屉内简单的文本搜索（菜名或小类），但不需要理解任何复杂分类规则。

### 5.2 餐品卡

卡片标题下显示路径标签：

> 中国菜 / 川菜 / 火锅麻辣烫 / 麻辣烫

默认停用的世界菜卡要清楚显示“未启用”；点击现有“启用此餐”即可加入起卦候选池。

### 5.3 编辑与新增

编辑 / 新增卡片中新增四个级联字段：版图、菜系、大类、小类。选择“自定义”后显示文本输入，可保留用户的本地自定义分类。原有餐品属性和删除 / 停用 / 启用行为不变。

## 6. 可访问性与移动端

- 所有筛选器均使用 `<label>` 与原生 `<select>` / `<input>`；
- “结果数量”和“当前筛选路径”通过已有 `aria-live` 区域提示；
- 筛选栏在小屏上纵向排列，按钮最小高度 44px；
- 菜单抽屉现有粘性关闭头部保持不变；
- 空筛选结果给出“清除筛选”的可用操作，不显示空白列表。

## 7. 测试与验收

新增或更新测试至少覆盖：

1. 默认菜单总数不少于 210，且中国菜默认启用数不少于 140、世界料理默认停用数不少于 70；
2. 每个默认餐品均有有效四级分类；
3. 默认起卦候选中没有未启用世界料理；
4. 启用世界料理模板后，它可参与与餐别 / 地点匹配的 `chooseMeal()`；
5. 旧版无分类本地菜单可正常读取且自动补齐回退分类；
6. 菜系 / 大类 / 状态筛选结果正确，清除筛选恢复完整列表；
7. 新增或编辑自定义分类不破坏菜单持久化；
8. 现有 64 卦、起卦、食录、菜单编辑与持久化测试继续通过。

浏览器验收：

- 首页保持没有菜系选择控件；
- 初始起卦只能从已启用中餐候选池中选择；
- 打开“整饬食单”后可按中国菜 / 世界料理、菜系、大类和状态筛选；
- 默认世界菜清楚可见但停用，启用一项后可在后续起卦中出现；
- 375px 与桌面均无横向溢出，抽屉关闭、Esc、焦点锁定仍可用。

## 8. 不在范围内

- 不增加首页菜系偏好或新的起卦输入；
- 不接入外卖平台、地图、餐厅库存或全球菜品真实性 API；
- 不替用户自动启用世界菜；
- 不删除用户旧菜单或食录。
