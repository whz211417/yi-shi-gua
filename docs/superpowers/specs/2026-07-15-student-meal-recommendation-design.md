# 学生就餐双层推荐模型设计

**日期：** 2026-07-15  
**状态：** 用户已确认方案，待实施计划审阅

## 问题与目标

现有系统把大量具体菜品作为直接起卦结果。它适合整理食单，却不适合学生日常决策：食堂不保证供应某一道菜，单道菜也不构成一顿可执行的饭。

新模型将“目录”与“推荐”分离：

- **目录层**保留 39 个菜系与 1,039 项具体菜品，用于浏览、启用外出偏好与记录；
- **推荐层**按去处生成现实可执行的就餐方案；
- 卦象提供透明的饮食倾向，天气、餐别、近期食录与去处决定最终排序。

首页仍只保留数起卦，不增加菜系或食堂筛选。

## 1. 去处与推荐单位

### 食堂

选择“食堂”时，结果以一份可替换的**餐盘结构**为骨架，并根据卦象给出 2 至 4 道常见、可在多数大学食堂自选窗口寻找的**具体菜建议**。它不强迫用户必须找到其中某一道菜：若当天无供应，用户可按同一蛋白、烹饪法或口味标签，从同窗口替换。

餐盘结构由窗口类型、主食、蛋白、蔬菜、汤/饮品与可选补充组成；其中主食、蛋白、蔬菜是完整正餐的必要部分，甜品、咖啡、炸物不可单独构成推荐结果。

首版采用通用大学食堂模型，所有项目标为“通用模板”，不宣称暨大必然供应。可用窗口与餐型如下：

1. 自选快餐：一荤两素 + 米饭 + 汤；两荤一素 + 米饭；素菜/豆制品组合。
2. 家常小炒：肉类或豆制品小炒 + 青菜 + 米饭。
3. 盖浇饭：肉类/蛋类/豆制品盖饭 + 汤或小菜。
4. 粉面档：汤粉、汤面、拌面、炒粉/炒面 + 蛋/豆制品/小菜。
5. 麻辣烫/冒菜：蔬菜、菌菇、豆制品与蛋白为主，搭配米饭/粉面。
6. 饺子馄饨：饺子、馄饨、云吞 + 小菜或汤。
7. 炒饭炒面：蛋炒饭、鸡肉炒饭、炒河粉、炒面 + 蔬菜/汤。
8. 铁板饭/煲仔饭：肉类主菜 + 蔬菜 + 米饭。
9. 汤饭汤面：汤泡饭、热汤粉、热汤面，适合雨天、风大或偏冷。
10. 早餐点心：粥、包点、肠粉、鸡蛋、豆浆、面包；早餐应补足蛋白。
11. 炸鸡/汉堡快餐：鸡排、炸鸡、汉堡等必须搭配蔬菜或汤，天气晴热时降权。
12. 轻食/便利补餐：三明治、饭团、关东煮、酸奶、玉米、鸡胸等，适合赶时间；不得只推荐饮品或甜品。

“食堂”结果给出 1 个优先方案、2 至 4 道具体菜建议和 2 个同类替代方案，使用户能按当日窗口实际供应替换。具体菜建议应为高频通用菜，不绑定某学校或档口，例如：

- 家常均衡：番茄炒蛋、土豆烧鸡、青椒肉丝、清炒时蔬；
- 温润热食：冬瓜排骨汤、萝卜牛腩、菌菇鸡汤、热汤面；
- 清爽轻食：白灼/清炒时蔬、香菇青菜、蒸蛋、鸡丝拌面；
- 耐饱补给：宫保鸡丁、黑椒牛柳、肉末茄子、鸡腿饭。

这些菜由卦象饮食标签和天气共同选出；它们是“优先寻找的具体菜”，不是虚构的当日供应承诺。

### 外出

选择“外出”时，先从用户启用的外出菜系抽出一个菜系，再从该菜系的已确认或通用目录中给出 2 至 4 个具体小类菜品。示例：

- 日料：拉面、盖饭、寿司；
- 潮汕菜：牛肉火锅、粿条汤、卤味饭；
- 泰餐：打抛饭、冬阴功面、咖喱饭；
- 川菜：盖饭、小炒、冒菜。

外出结果必须提示“按附近实际店铺供应选择”，不虚构门店、价格或距离。世界料理须仍由用户在目录内主动启用后，才具备外出推荐资格。

## 2. 天气模型

天气选项更换为六种体感场景，属于排序权重，不是硬性禁令：

| 场景 | 食堂排序倾向 | 外出排序倾向 |
|---|---|---|
| 晴热 | 清爽、汤粉、拌面、蔬菜多、少油炸 | 减少浓重炸物与远距离热锅 |
| 晴暖 | 均衡餐 | 正常排序 |
| 阴凉 | 热饭、热菜、汤羹加权 | 正常，可偏热食 |
| 雨天 | 汤面、汤粉、盖饭；近距离食堂加权 | 降低外出权重 |
| 风大 | 热食、饭面汤加权 | 降低远距离轻食 |
| 寒冷/雨雪 | 汤锅、热饭、热面与足量蛋白加权 | 火锅、炖菜、热汤类加权 |

旧天气值在本地存储和现有 UI 事件中应防御性映射到新天气值，避免历史配置报错。

## 3. 卦象到饮食倾向的透明映射

系统不声称卦象能精确算出某道菜。它使用一套可见、稳定、娱乐用途的映射，作为候选排序的一部分：

- 坎：温润、汤羹、热食；
- 离：鲜香、明快、色泽丰富；
- 震：快捷、易获得、适合时间紧；
- 巽：轻盈、蔬菜、清爽；
- 艮：稳妥、少量、不过度折腾；
- 兑：适度改善、分享、小食可作配；
- 乾：足量蛋白、耐饱；
- 坤：均衡、家常、餐盘完整。

每次结果必须展示三段理由：

1. **卦象取向**：本卦或主要卦象如何产生饮食倾向；
2. **现实修正**：餐别、去处、天气、近期食录如何改变候选排序；
3. **落地替代**：若首选窗口没有供应，可使用哪两类同餐型方案替换。

说明必须标注为“娱乐性饮食提示”，不得声称医疗、营养诊断、天气实况或传统术数的确定性结论。

## 4. 数据边界与持久化

新增独立的通用食堂餐型数据源，不污染现有 1,039 项菜系目录。推荐器接受统一候选接口：

- `kind: 'canteen-plan' | 'outing-cuisine'`；
- 场景、完整餐标记、窗口类型、基础营养标签、天气偏好、卦象倾向；
- 外出候选引用已有的启用菜系与具体小类；
- 食堂模板默认可用，但标注为“通用模板”；
- 旧菜单、食录、世界料理启用状态与目录迁移规则不被重置。

## 5. 结果呈现与验收

起卦结果改为：标题、就餐方式、优先方案或菜系、2 至 4 个可替换小类、三段理由、天气提示、娱乐性说明。确认后写入食录的记录应保存实际选择的结果名称与所属就餐场景。

验收要求：

1. 食堂永不直接输出单独甜品、咖啡或小食作为完整正餐；
2. 食堂结果总能表达完整餐型，提供 2 至 4 道与卦象/天气匹配的具体高频菜建议，并附同类替代；
3. 具体菜建议明确标为通用可替换建议，不声称学校当日必然供应；
4. 外出只从已启用的外出菜系中产生，且给出多项小类选择；
5. 六种天气均有可解释的排序影响；
6. 每次结果均展示卦象、现实条件、替代方案三段解释；
7. 64 卦、确认/重试、食录、旧菜单迁移、目录浏览均不回归；
8. 纯函数、旧存储模拟、桌面/移动浏览器与 GitHub Pages 全部验收。
