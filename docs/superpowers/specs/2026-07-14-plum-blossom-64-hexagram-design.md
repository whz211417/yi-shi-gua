# 易食卦「数时合参 · 六十四卦」专业化设计规格

**日期：** 2026-07-14  
**状态：** 已确认方向，待用户审阅规格  
**定位：** 将现有泛化“食卦文案”升级为可复算、可解释、可视化的六十四卦数时合参系统；易学内容仅作文化与娱乐性叙事，饮食推荐仍受日常均衡规则约束。

## 1. 问题与目标

当前页面虽有 64 条泛化文字，但不展示真正的本卦、上下卦、动爻、变卦或起卦过程。用户输入的数和时辰只被混入一个内部 seed，用户无法核验，视觉上也仅见空状态装饰，容易被理解为随机抽签。

新版目标：

1. 固定用 **北京时间（Asia/Shanghai）** 读取农历月、农历日与十二时辰；
2. 将用户输入的 1–64 数、农历月日与时辰按公开规则计算本卦、互卦、动爻、变卦；
3. 用正确的六爻阴阳图、卦名、上下卦、卦序和动爻呈现结果；
4. 每次结果可复算：相同报数与相同北京时间必得相同卦象；
5. 卦象参与候选餐的确定性排序，但不越过餐别、地点、去重、蛋白和蔬菜等日常规则；
6. 将动效改为“六爻书成 → 动爻朱砂点定 → 变卦侧显”，避免空泛加载条与普通随机感。

## 2. 方法声明与边界

页面明确标注：

> 本页采用“数时合参起卦法”，依梅花易数常见的取数、取时、取余思路组合实现；不同传承的取数细节可能不同。本结果仅供文化体验与日常选择叙事，不作预测、医疗或命理承诺。

这不是声称唯一正统的梅花易数公式，也不采集八字、出生信息或位置。

## 3. 时间与农历来源

### 3.1 时区

- 一律使用 `Asia/Shanghai`，不随访问者设备时区变化；
- 页面显示用于计算的北京时间和农历值，便于复算；
- 即使海外用户访问，卦象仍遵循同一时区规则。

### 3.2 农历

- 使用浏览器原生 `Intl.DateTimeFormat('zh-Hans-CN-u-ca-chinese', { timeZone: 'Asia/Shanghai' }).formatToParts()`；
- 解析农历月、农历日，并展示闰月标记；
- 闰月计算时沿用该月数字，显示层保留“闰”字；
- 若浏览器不支持中国农历日历，不静默降级为公历。应在起卦区域显示明确提示“当前浏览器不支持农历起卦，请使用新版 Chrome、Edge 或 Safari”，并阻止起卦。

### 3.3 十二时辰

北京时间小时映射：

| 时段 | 时辰 | 序号 |
|---|---:|---:|
| 23:00–00:59 | 子 | 1 |
| 01:00–02:59 | 丑 | 2 |
| … | … | … |
| 21:00–22:59 | 亥 | 12 |

即：`hourBranch = Math.floor(((beijingHour + 1) % 24) / 2) + 1`。

## 4. 数时合参公式

设：

- $n$：用户报数，范围 1–64；
- $m$：农历月（1–12，闰月沿用月数）；
- $d$：农历日（1–30）；
- $h$：十二时辰序号（子=1，…，亥=12）。

八卦余数顺序固定为：

| 余数 | 卦 | 二进制爻（自下而上，阳=1） |
|---:|---|---|
| 1 | 乾 ☰ | 111 |
| 2 | 兑 ☱ | 110 |
| 3 | 离 ☲ | 101 |
| 4 | 震 ☳ | 100 |
| 5 | 巽 ☴ | 011 |
| 6 | 坎 ☵ | 010 |
| 7 | 艮 ☶ | 001 |
| 0 / 8 | 坤 ☷ | 000 |

计算：

$$
上卦 = (n+m+d) \bmod 8
$$

$$
下卦 = (n+h) \bmod 8
$$

$$
动爻 = (n+m+d+h) \bmod 6
$$

余数为 0 时，上/下卦按 8（坤）处理，动爻按 6 处理。动爻从下往上计为初、二、三、四、五、上。

### 4.1 三类卦

- **本卦：** 下卦三爻在下、上卦三爻在上，匹配完整六十四卦映射表；
- **互卦：** 本卦第 2、3、4 爻组成下互卦，第 3、4、5 爻组成上互卦；
- **变卦：** 将本卦对应动爻阴阳翻转后，匹配完整六十四卦映射表。

所有卦名必须为标准通行名称与文王卦序，例如“地山谦（第 15 卦）”“水火既济（第 63 卦）”；不得继续把泛化 `FOOD_ORACLES` 条目冒充真实卦象。

## 5. 数据与推荐架构

### 5.1 纯函数边界

新增纯函数（名称可按现有代码风格微调）：

- `beijingCalendarParts(date)`：由 `Date` 得到北京时间农历月日、闰月、时辰序号及显示文本；
- `trigramFromRemainder(value)`：处理 0→8 余数规则；
- `hexagramForTrigrams(upper, lower)`：从 64 卦表取本卦；
- `deriveMutualHexagram(lines)`：从六爻推导互卦；
- `deriveChangedHexagram(lines, movingLine)`：翻动爻推导变卦；
- `deriveDivination(reportNumber, calendarParts)`：返回完整可序列化结果。

`deriveDivination` 接受可注入的日历参数，因此 Node 测试不依赖本机时区或 ICU 差异。

### 5.2 六十四卦表

在 `assets/data.js` 中用固定数据表存储 64 个条目，每项至少含：

```js
{
  number: 63,
  name: '水火既济',
  upper: '坎',
  lower: '离',
  lines: [1, 0, 1, 0, 1, 0]
}
```

`lines` 顺序始终为“自下而上”。数据表需覆盖 1–64 全部标准卦名、卦序、上下卦，且 64 个六爻编码唯一。

原 `FOOD_ORACLES` 可删除或只保留为餐饮提示素材；页面展示的卦名、卦序与卦图必须完全来自新表。

### 5.3 餐食选择

起卦先产生 `divination`，然后以本卦卦序、变卦卦序、动爻、用户报数与餐别/地点/天气构成确定性 seed，交给现有 `chooseMeal()` 的平分决胜逻辑。

- 现有资格筛选、历史去重和均衡评分不可删除；
- 卦象只影响通过规则筛选后的稳定排序；
- 页面明确写“餐食仍优先遵循餐别、地点与日常均衡规则”。

## 6. 页面与文案

### 6.1 起卦前

替代空的三横线装饰，显示低对比六爻底纹和一句说明：

> 报数与北京时间农历时辰合参，起本卦、互卦与变卦。

起卦区域增加“按北京时间”提示，不增加额外输入项。

### 6.2 结果卡

结果卡固定结构：

```text
第 32 数 · 北京时间 11:24 · 农历五月十九 · 午时

                 ☵
           水火既济
                 ☲

本卦  水火既济 · 第 63 卦
互卦  火水未济
动爻  九三
变卦  水山蹇

数时合参：32 + 农历五月十九 + 午时
餐签：鸡丝汤面加青菜
```

实际卦图必须由六行爻绘制，不依赖某个字体是否有 Unicode 卦象。阳爻为连续横线，阴爻为中断横线；本卦动爻有朱砂侧标；变卦以较小、低对比的六爻图置于右侧或移动端的下方。

在本卦信息和餐签之间显示一行简短说明：

> 本卦用于此次餐签排序；餐食仍以餐别、地点与日常均衡规则为先。

### 6.3 动效

保留已完成的“定数→聚墨→落印”克制风格，但在 `ink` 阶段让本卦六爻由下至上以笔画方式依次书成；在 `seal` 阶段标记动爻、浮出变卦侧签、再显示餐签。

- 六爻书成不是加载条：每爻是静态传统阴阳爻，并在完成后保留；
- 动爻强调仅限一个朱砂点/短侧标；
- 禁止持续旋转、粒子、爆炸、3D 翻转和声音；
- `prefers-reduced-motion` 下直接显示完整本卦、互卦、动爻、变卦和餐签。

## 7. 测试与验收

### 7.1 数据与算法测试

新增测试至少覆盖：

1. 64 卦表恰有 64 条、卦序 1–64 完整、卦名和六爻编码唯一；
2. 八卦余数 `0` 正确映射为坤，动爻余数 `0` 正确映射为上爻；
3. 示例：$n=32,m=5,d=19,h=7$ 得本卦“地山谦（15）”、动爻“三”、变卦“坤为地（2）”；
4. 互卦由第 2–4 与第 3–5 爻正确推导；
5. 相同可注入日历参数和报数返回完全相同的结果；
6. 翻转指定动爻只改变一条爻；
7. 现有推荐、存储、菜单数据测试全部继续通过。

### 7.2 浏览器验收

在 Chrome 的 1440×900、375×812：

- 起卦前和结果后都可看到真实六爻而非空横线；
- 本卦、互卦、动爻、变卦、农历月日、北京时间时辰均可见；
- 改变报数会影响卦象；同一时刻还原报数可复现同一结果；
- `ink` 阶段仅书成六爻，`seal` 阶段才出现餐签；
- 无横向溢出，移动端变卦改为纵向排布；
- 菜单、食录、确认、重试、减少动效和浏览器控制台均无回归。
